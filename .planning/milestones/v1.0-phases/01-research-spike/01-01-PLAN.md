<objective>
Empirically verify whether the d2core.com `function-planner-queryplan` Cloud Function response includes skill allocation data, and document the exact API endpoint, request format, and response JSON schema.

Purpose: Resolve the #1 architectural blocker for the entire project — the web_parser (Phase 3) architecture depends entirely on whether skills data exists in the API response. If skills are present, Phase 3 uses a simple direct-HTTP call (~50 lines of Rust). If absent, Phase 3 requires a headless browser or secondary API discovery, adding substantial complexity and a ~150 MB shipping dependency.

Output: `SPIKE-FINDINGS.md` containing the verdict, exact endpoint URL, full response JSON, skills data findings, architecture decision with rationale, and at least 2 bd= test vectors for use as Phase 3 pinned test data (PARSE-07).
</objective>

  <read_first>

  </read_first>

  <what-to-do>
    This investigation cannot be automated — it requires a live browser session and network access to d2core.com. Follow these exact steps:

    STEP 1 — Find 2 build URLs:
    - Go to https://www.d2core.com/d4/planner and find or enter public build URLs
    - Good sources: Reddit r/diablo4 "share your build" posts, D4 Discord pinned builds, or use the d2core search/explore feature
    - Record at least 2 bd= values (e.g. bd=1MWy, bd=AbCd) — ideally different classes (e.g. Necromancer + Barbarian)
    - Verify each build URL actually loads in the browser before proceeding

    STEP 2 — Open DevTools and capture the API call:
    - Open Chrome or Firefox
    - Press F12 to open DevTools and go to the Network tab
    - Enable "Preserve log" checkbox
    - Navigate to the first build URL: https://www.d2core.com/d4/planner?bd=XXXX
    - In the Network filter bar, type: queryplan
    - If no results, also try: cloudfunctions, web?env, or filter by Fetch/XHR
    - The matching request is a POST — click it

    STEP 3 — Record request details (for EVERY captured request):
    - Full URL (from the Headers tab)
    - HTTP method
    - ALL request headers — including Authorization, Content-Type, Origin, Referer, Cookie
    - Raw request body/payload (look in "Payload" or "Request" tab)

    STEP 4 — Copy FULL response body (CRITICAL: do not truncate):
    - Click the "Response" tab (NOT "Preview")
    - Right-click the request → "Copy" → "Copy response" to get the full text
    - The response_data field contains a JSON-ENCODED STRING inside JSON — you MUST paste the response_data VALUE into a separate JSON formatter (https://jsonformatter.org/) to see the inner schema
    - Record all field names in the decoded inner JSON, especially whether variants[N] contains any key with "skill" in the name

    STEP 5 — Check for skills data:
    - Search the full decoded JSON for any of these keys (case-insensitive): skills, skill, skilltree, skillAllocations, skill_allocations, skillPoints, activeSkills, passiveSkills, buildSkills
    - Record your finding: YES (skills present) / NO (only gear + paragon) / PARTIAL (some but not all)

    STEP 6 — If NO skills in API, check the DOM:
    - Switch to the Elements tab in DevTools
    - Find the skill tree section visually on the page
    - Inspect individual skill nodes: record the element type, CSS class pattern, and any data-* attributes
    - In the Console tab, run: JSON.stringify(window.__NEXT_DATA__ || window.__NUXT__ || window.store || "not found")
    - Note what you find

    STEP 7 — Repeat Steps 2–5 for the second build URL

    STEP 8 — Check auth requirement:
    - Look at request headers from Step 3 — is there an Authorization header or Firebase auth token in the request body?
    - Record YES/NO and the exact token format if present

    RESUME SIGNAL — Paste all captured data into your reply:
    - The exact POST URL from the network capture
    - All request headers
    - The raw request body JSON
    - The raw response body JSON (full, not truncated)
    - The decoded inner JSON (after parsing response_data)
    - Your YES/NO/PARTIAL verdict on skills data
    - If NO: the DOM element structure for skill nodes
    - The same information for the second build URL (bd= value)
  </what-to-do>
